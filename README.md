# Требования
## Сервер
- Взаимодействие с БД через ORM
- Реализация возможности создания, чтения, редактирования и удаления (CRUD). 
- Реализация возможности фильтрации.
- Предоставление веб-интерфейса (API) для клиента.

## Клиент
- Ввод данных (поиск, создание, фильтрация данных)
- Отображение данных
- Обращение к серверу через API
- Формирование дашборда из полученных данных

# API
Для клиентских обращений, сервер использует порт `8337` и протокол `HTTP`

## Эндпоинты

### /account

#### /account/create (POST)
Метод для регистрации нового пользовательского аккаунта.

**Body-параметры**
- `login` (string): Пользовательское имя, по которому тот идентифицируется.
- `password` (string): Пароль, введенный пользователем. Пересылается открытым образом.
- `role` (string): Идентификатор роли пользователя. Допустимые: USER, PARTNER, ADMIN.

**Авторизация**: Не требуется.

**Минимальная требуемая роль**: Не требуется 

**Возвращаемые параметры при успешном запросе**:
- `account` (account): Новосозданный объект.
- `refreshToken` (string): Токен для перегенерации Access-токена при истечении того.
- `accessToken` (string): Токен, который должен передаваться в хэдере Authorization в формате Bearer {Token} (без скобок).


#### /account/read (GET)
Метод для получения объекта.

**Path-параметры**
Параметры, которые указываются после пути эндпоинта следующим образом `/{param}`
- `id` (int): Числовой идентификатор, получаемого пользователя.

**Авторизация**: Bearer Access-Token

**Минимальная требуемая роль**: `USER`

**Возвращаемые параметры при успешном запросе**:
- `account` (account): Найденный объект.

#### /account/update (PUT)
Метод для обновления объекта.

**Body-параметры**

- `role` (string): Идентификатор роли пользователя. Допустимые: USER, PARTNER, ADMIN.

**Авторизация**: Bearer Access-Token

**Минимальная требуемая роль**: `ADMIN`
